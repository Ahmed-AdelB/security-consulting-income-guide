## Firmware & Embedded Security Analysis
### Firmware Analysis Workflow (High-Level)
- Identify firmware sources (vendor portal, field extraction, OEM kit).
- Capture firmware version, build date, and hardware compatibility.
- Preserve hashes and chain of custody for evidence.
- Validate signature and integrity checks.
- Extract firmware filesystem for static analysis.
- Inventory binaries, configs, and embedded credentials.
- Locate hard-coded keys or default passwords.
- Analyze update mechanism and rollback protections.
- Validate secure boot configuration and enforcement.
- Review cryptographic library usage and entropy sources.
- Map privilege boundaries and service accounts.
- Identify debug or maintenance interfaces exposed by default.
- Assess logging and telemetry safety for OT/IoT usage.
- Document patchability constraints and maintenance windows.

### Firmware Analysis Checklist (Detailed)
- Confirm device model, hardware revision, and CPU architecture.
- Capture bootloader version and configuration.
- Check secure boot enablement flag.
- Verify boot chain signature validation.
- Confirm root of trust location (TPM, HSM, secure element).
- Identify firmware encryption mechanisms.
- Enumerate filesystem partitions and mount points.
- Locate kernel version and build settings.
- Extract init scripts and service definitions.
- Enumerate open network services in default config.
- Check for default credentials in config files.
- Search for embedded API keys or certificates.
- Validate TLS configuration for cloud connections.
- Inspect OTA update package format and signature.
- Validate update rollback protections and anti-downgrade.
- Confirm integrity verification on update apply.
- Evaluate update transport security (HTTPS, MQTT, proprietary).
- Review key rotation and certificate renewal behavior.
- Identify unsafe debug flags or developer modes.
- Inspect logging verbosity and sensitive data leakage.
- Map privilege separation between processes.
- Review device identity provisioning workflow.
- Check for insecure boot parameters.
- Identify kernel modules with known CVEs.
- Validate hardware watchdog behavior and fail-safe modes.
- Check for exposed JTAG or UART pins.
- Review secure diagnostics protocols and access control.
- Validate memory protection features (ASLR, NX, stack canaries).
- Inspect crypto implementation for deprecated algorithms.
- Check for hard-coded hostnames or IPs.
- Review privilege escalation paths in startup scripts.
- Inspect IPC mechanisms and access controls.
- Identify insecure file permissions on sensitive files.
- Check for unpatched open-source components.
- Verify SBOM completeness and accuracy.
- Confirm sandboxing or containerization usage.
- Review device factory reset behavior and data wipe.
- Assess local storage encryption for device data.
- Evaluate backup and restore mechanisms.
- Check for unsafe remote management features.
- Confirm time sync and certificate validation logic.
- Inspect secure logging for regulatory evidence requirements.
- Review event retention limits in constrained devices.
- Verify firmware signing key storage protections.
- Check for build artifacts revealing source paths or secrets.
- Map external API dependencies and auth mechanisms.
- Validate fail-safe behavior for firmware update failure.
- Review OTA staging environment and pre-prod validation.
- Confirm incident response procedure for firmware compromise.
- Document patch deployment cadence and customer notice windows.
- Define compensating controls for unpatchable firmware.
- Capture evidence screenshots and hashes for reporting.
- Align findings to CWE/CVE references where possible.
- Provide risk rating aligned to operational impact.

### Secure Boot and Key Management Focus
- Verify secure boot chain from ROM to bootloader to kernel.
- Confirm secure element or HSM is used for key storage.
- Validate key provisioning workflow and separation of duties.
- Ensure private keys are not present in firmware images.
- Review key rotation policy for fleet devices.
- Confirm support for revocation and blacklist updates.
- Validate attestation or device identity proof where applicable.

### OTA Update Security Checks
- Validate update packaging and signature verification.
- Confirm encrypted transport for update delivery.
- Review update server authentication and authorization.
- Verify integrity checks before and after update apply.
- Validate rollback protections to prevent downgrade attacks.
- Review staged rollout and canary deployment mechanisms.

### Hardware Interface and Physical Access Checks
- Identify exposed JTAG/SWD/UART headers.
- Check for physical port locks or tamper seals.
- Validate secure diagnostics access control.
- Assess susceptibility to firmware extraction attacks.
- Review EEPROM and flash access protections.

### Embedded Protocol and Bus Analysis
- Review CAN/LIN bus security assumptions for vehicles.
- Evaluate ISO 11783 (ISOBUS) message authentication gaps.
- Validate SecOC usage for message authentication where supported.
- Map insecure OT protocols (Modbus, DNP3, Ethernet/IP).
- Identify insecure remote management protocols (Telnet, HTTP).

### Firmware and Embedded Deliverables
- Firmware analysis report with vulnerability list and evidence.
- Secure boot assessment summary with remediation steps.
- OTA security architecture review document.
- Device identity and key management recommendations.
- Firmware update risk register with operational impact.
